## 4. 아키텍처

```
브라우저 (사용자)
   ↓
Next.js on Vercel
   ├─ /             페이지 (입력 + 결과 표시)
   ├─ /api/lookup   POST — 8개 병렬 조회
   └─ /api/cron/warmup  Cron 10분 — Playwright로 3개 세션 워밍업
                          ↓ 쿠키 저장
                    Upstash Redis (Vercel Marketplace)
                          ↑ 쿠키 읽기
   └─ /api/lookup (하이브리드 카드)
```

### 핵심 설계 결정

**즉석 워밍업 X, 주기 워밍업 O**
- 이유: 사용자 요청마다 Playwright 실행하면 15~30초 cold start. 10분 주기 워밍업으로 쿠키 KV 캐시 → 사용자 응답 시간 2~5초 유지.

**Stateless API + KV 1개**
- DB 없음. 조회 이력 없음. KV에는 카드사별 쿠키 문자열만 (TTL 15분).

**Promise.allSettled**
- 한 카드사 실패가 다른 카드사 결과를 막지 않음. 부분 실패도 나머지는 표시.

**per-module timeout 10초**
- 느린 카드사가 전체 응답 지연시키지 않도록.

### 모듈 책임

- **`cards/*.ts`**: 각 카드사 한 곳만 책임. 공통 시그니처 `check(bizNo) → CardResult`. 5개는 자체 HTTP, 3개는 KV 쿠키 사용.
- **`warmup/*.ts`**: Playwright 시나리오. Cron에서만 호출.
- **`lib/kv.ts`**: KV 접근 캡슐화 (`getCookie('woori')`, `setCookie('woori', value)`).
- **`app/api/lookup/route.ts`**: 8개 모듈 `Promise.allSettled`로 병렬 호출.
- **`app/api/cron/warmup/route.ts`**: Bearer 인증 후 3개 워밍업.

## 7. API 플로우

### `POST /api/lookup`

```
Request:  { bizNo: "4784001474" }
Response: { bizNo, results: CardResult[], totalMs }

1. validate(bizNo) — 10자리 숫자 확인 (아니면 400)
2. const modules = [bc, lotte, samsung, hyundai, kb, woori, hana, shinhan]
3. const settled = await Promise.allSettled(
     modules.map(m => m.check(bizNo))
   )
4. 결과 변환 후 반환
```

### `GET /api/cron/warmup`

```
Headers: Authorization: Bearer {CRON_SECRET}

1. Bearer 인증 실패 → 401
2. Playwright + @sparticuz/chromium 실행
3. 3개 병렬 워밍업:
   - woori-warmup, hana-warmup, shinhan-warmup
4. 각 쿠키 → KV에 저장 (TTL 900초 = 15분)
5. 응답: { success: 3, failed: 0, ... }
```

### 하이브리드 카드 폴백 로직

```
check(bizNo):
  cookie = await kv.get(cardName)
  if !cookie: return { status: 'error', error: 'session_not_ready' }
  
  result = await fetch(API, { Cookie: cookie })
  if 세션만료_감지: return { status: 'error', error: 'session_expired' }
  return parseResult(result)
```

즉석 워밍업은 하지 않음 — 느린 UX 회피. Cron이 다음 주기에 복구.

## 9. 에러 처리 정책

| 상황 | 사용자 화면 | 로그 |
|------|-----------|------|
| 카드사 사이트 점검 | ⚠️ "일시 오류" | HTTP status/timeout |
| 타임아웃 (10초) | ⚠️ "응답 지연" | `timeout` |
| 세션 만료 (하이브리드) | ⚠️ "세션 복구 중" | 만료 카드사 |
| Cron 실패 (KV 비어있음) | ⚠️ "준비 중" | warmup 실패 원인 |
| 사업자번호 형식 오류 | 필드 아래 문구 | — |
| API 전체 장애 | 배너 | 스택 |

**원칙**: 한 카드사 실패가 다른 결과를 막지 않음. 에러는 해당 카드에만.

## 14. 리스크 / 한계

- **Cron 실패 시 3개 카드사 응답 불가**: Playwright가 카드사 페이지 구조 변경에 취약. 워밍업 시나리오 유지보수 필요.
- **우리/하나/신한 페이지 구조 변경**: 셀렉터 깨지면 Cron 실패 → 수동 보정 필요.
- **Vercel Pro 비용**: 월 $20. 사내 도구 비용 정당화 필요.
- **약관 검토**: 각 카드사 이용약관의 자동조회 금지 조항 여부 확인 권장. 내부 업무 보조 용도라 실무적 리스크는 낮음.
- **IP 차단 위험**: 단건 조회라 배치보다 낮음. Vercel IP는 다수 이용자가 공유 → 다른 사용자 활동의 영향 받을 수 있음.
